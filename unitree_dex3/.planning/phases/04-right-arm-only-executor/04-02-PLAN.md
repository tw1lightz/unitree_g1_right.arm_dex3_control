<objective>
Insert two-stage trajectory validation at the very top of `trajectoryCallback`, immediately after the existing empty-`joint_names` guard and before any motion logic. Stage 1 logs `RCLCPP_WARN` and strips foreign (non-right-arm) joint columns into a local working copy. Stage 2 verifies the stripped working set contains all 7 right-arm joints; otherwise `RCLCPP_ERROR` and `return` without publishing any `LowCmd`. The original `msg` stays read-only (D-05). The waypoint loop's inner read of trajectory columns is rewired to consume the working copy, so foreign columns are never indexed. No changes to the publish-write side yet — that is Plan 04-03's job.
</objective>

<threat_model>
## Security / Safety Threats Considered

- **Stripping foreign columns mid-trajectory**: stripping happens before any motion logic, no master change, and no `LowCmd` publish. Safe at all times because the executor's prior state (whatever master value the previous trajectory left) is untouched on rejection.
- **`joint_limits_.at()` throw on unknown name**: stage 2 guarantees every entry in `right_arm_columns_in_msg` resolves to a right-arm name. Right-arm names are populated by `joint_limits_` at startup (URDF parse) for all REVOLUTE joints. The throw is unreachable in this code path. If URDF parsing produced an empty `joint_limits_` (warning at L111 / L114), the same `.at()` would throw at the OLD inner loop too — pre-existing risk, not introduced here.
- **`isRightArmJoint` linear scan vs hash set**: 7-element linear scan is O(7) per joint name, called once per row in `msg->joint_names` (typically 7-21). Total worst case ~150 string comparisons per `trajectoryCallback`, called maybe once per second of robot wall-clock. Negligible compared to the 250 Hz publish loops. No DoS / hot-loop concern.
- **Log-injection via foreign joint names**: foreign names come from a ROS 2 trajectory message published by the planner (or in a synthetic test, by `ros2 topic pub`). `RCLCPP_WARN` formats them as a `%s` argument; ROS 2 logging treats this as plain text. No format-string vulnerability because the format specifier is fixed.
- **No `LowCmd` published on rejection**: critical safety property — the master switch stays at whatever value the previous frame published, so the robot doesn't experience a sudden authority change due to a rejected trajectory.

No HIGH severity threats. LOW severity log-volume concern (a malicious trajectory with 1000 foreign joint names would produce a long single WARN line); negligible in practice.
</threat_model>
