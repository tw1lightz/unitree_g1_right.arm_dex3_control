# Plan 01-06: Fix gracefulRelease control-authority steal jerk

## Why

User-reported regression during Plan 01-04 human-verify checkpoint:
> "当我 Ctrl+C 退出 executor 后,它会速度很快的把两只手都向前伸,然后才缓慢回到 running mode 的手臂位置."

Plan 01-04 added `gracefulRelease()` which is called from `main()` AFTER the spin loop exits. Two issues compound:

### Root cause 1 — `gracefulRelease()` re-grabs control authority

`gracefulRelease()` at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:138-144` runs a 150-step ramp where step 0 publishes:

```cpp
const double value = (1.0 - 0.0/150) * 1.0;  // = 1.0
release_cmd.motor_cmd[JointIndex::kNotUsedJoint].q = static_cast<float>(value);
cmd_pub_->publish(release_cmd);
```

By the time `gracefulRelease()` runs, `trajectoryCallback`'s **own** end-of-trajectory ramp at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:249-260` has already driven `kNotUsedJoint.q` from 1.0 down to 0.0 — the body controller is already in charge and the arm is already back to standing. `gracefulRelease()`'s step-0 publish then **slams the master switch back to 1.0**, planner re-grabs authority, and because the freshly-default-constructed `release_cmd` has every other `motor_cmd[i].q = 0` (and `kp = 0`, `kd = 0`), the arm jerks toward whatever pose `q = 0` corresponds to under arm_sdk's default behavior — observed as "both hands forward". Then the 150-step ramp slowly releases authority and the body controller takes over again.

The trajectory-end ramp (line 249-260) does NOT have this problem because it is preceded by a fully-populated `cmd_msg` (kp=60, kd=1.5, q=current trajectory point). The first step-0 ramp message (kp=0+q=0) immediately slackens the joints; the master switch then fades smoothly. There is no "steal" because authority was never released between the populated `cmd_msg` and the ramp.

### Root cause 2 — `trajectoryCallback` is synchronous and non-interruptible

If the user presses Ctrl+C **mid-trajectory**, the atomic flag is set immediately, but `trajectoryCallback` keeps running synchronously until it has finished all waypoints + hand-close + its own ramp. Only then does `main()` see the flag, exit the spin loop, and run `gracefulRelease()` (which then triggers root cause 1). So the user-visible behavior on a mid-trajectory Ctrl+C is "robot finishes the planned motion, then jerks, then settles" — the opposite of "stop now and release smoothly".

## Constraint analysis

The cleanest minimal fix is to:

1. **Delete `gracefulRelease()`** entirely. The trajectory-end ramp at line 249-260 is the ONLY graceful release we ever needed; running it twice is what creates the steal.
2. **Make `trajectoryCallback`'s waypoint loop check `g_shutdown_requested`**, breaking out at the next loop iteration. The existing trailing code path (hand close + 1 s ramp) then runs from the current trajectory point, producing exactly the behavior the user wanted: stop near current pose, smooth release.
3. **Keep** the signal handler / `uninstall_signal_handlers()` / atomic flag / `SingleThreadedExecutor` polling loop in `main()`, because the polling loop is what allows `main()` to return cleanly after `trajectoryCallback` finishes (or when Ctrl+C is pressed in idle state).
4. **Keep** the destructor's instantaneous `q = 0.0` fallback. In the idle-state-Ctrl+C path, the body is already in control (last trajectory's ramp already finished) so this final publish is a redundant but harmless reaffirmation of the body-controller authority.

Idle-state Ctrl+C path after this fix:
- main loop sees flag, exits without publishing anything
- destructor runs, publishes one `kNotUsedJoint.q = 0.0` LowCmd (harmless reaffirmation)
- process exits

Mid-trajectory Ctrl+C path after this fix:
- `trajectoryCallback`'s next loop iteration sees the flag, breaks
- existing trailing code runs: hand close (1 s) + trajectory-end ramp (1 s)
- `trajectoryCallback` returns, main loop sees flag, exits, destructor runs

No new code paths, no new logging, no new state.

## Decision

Three surgical edits in `joint_trajectory_executor.cpp`:

### Edit 1 — delete `gracefulRelease()` method

Remove the entire 26-line block at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:126-149`.

### Edit 2 — remove `gracefulRelease()` call from `main()`

Remove the 3-line block at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:280-282`. The polling loop, signal handler installation, and `uninstall_signal_handlers()` all stay.

### Edit 3 — add shutdown check to `trajectoryCallback` waypoint loop

Modify the for-loop at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:199` from:

```cpp
for (size_t i = 0; i < msg->points.size(); ++i) {
```

to:

```cpp
for (size_t i = 0; i < msg->points.size() && !g_shutdown_requested.load(); ++i) {
```

That single condition addition lets the loop exit on the next waypoint after the user hits Ctrl+C, dropping straight into the existing hand-close + ramp tail. No other code in the function needs to change.

## Notes

This plan **reverts the design choice in Plan 01-04** (a separate `gracefulRelease()` method called from `main()`). Plan 01-04's analysis missed that `trajectoryCallback`'s end-of-trajectory ramp already covers the only scenario that requires a graceful release, and that running a second ramp from idle state inevitably steals authority from the body controller.

The lesson recorded for `key-decisions` in the SUMMARY: **never publish a LowCmd with `kNotUsedJoint.q = 1.0` from a state where the body controller is already in charge** — the master switch is a snap toggle, not a fade.
