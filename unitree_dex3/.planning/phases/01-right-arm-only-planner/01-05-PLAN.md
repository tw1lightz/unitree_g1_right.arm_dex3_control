# Plan 01-05: Unblock live verification

## Why

Plan 01-03 smoke test only reached the init log without sending a `/goal_pose`. The first goal_pose during user-driven live verification (Plan 01-03 + 01-04 checkpoint) surfaced two pre-existing defects that block `checkpoint:human-verify`:

1. **`Joint-order mismatch ... right_elbow_joint out of order at idx 22; right_wrist_pitch_joint out of order at idx 37`** — emitted as `RCLCPP_ERROR`, makes the operator think the planner is broken. **It isn't.** The very next block (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp:403-412`) does a `std::find` by **joint name** to fill `planning_positions`, so the index ordering of `joint_names_` (which is whatever `joint_state_publisher` emits) is irrelevant to functional correctness. The check is dead-weight noise.

2. **`Collision detected between right_elbow_link and right_wrist_roll_link`** → start = INVALID → OMPL aborts. These two links are **physically adjacent** in URDF (connected by `right_wrist_roll_joint`); their collision meshes geometrically overlap by construction. They MUST be skipped. The project's `collision_skip_pairs_` parameter defaults to empty (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp:62`) and `planner.launch.py` also defaults it to empty (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/launch/planner.launch.py:53`). git show `83b34a5` (project_init) confirms both have been empty since day one.

Both are pre-existing — not introduced by Plans 01-01..01-04. They were not in the Phase 1 RESEARCH inventory because the Phase 1 smoke test did not exercise the goal callback path. Surfacing them at human-verify is the system working as designed; we just need to fix them before declaring Phase 1 verified.

## Background

`kdl_chain_right` is built once in `init()` (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp:112`) by `kdl_parser::treeFromUrdfModel` + `getChain(base_link_, right_tip_, ...)`. Each `KDL::Segment::getName()` returns the **child link name** (verified by line 327 `planning_links.insert(seg.getName())` which is later used to filter collision pairs by link name). The chain does not include the base, so the parent of segment 0 is `base_link_` itself.

Therefore the canonical adjacent-link pair list for the right arm is:

```
(base_link_, segment[0].name)
(segment[0].name, segment[1].name)
...
(segment[N-1].name, segment[N].name)
```

For `base_link_ = "torso_link"` and `right_tip_ = "right_wrist_yaw_link"`, this is 7 pairs (one per joint in the chain).

## Constraint analysis

- The existing per-state `isInCollision` already accepts a `skip_pairs` vector (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp:628`) and matches by `link1:link2` or `link2:link1` (line 668-674). We do NOT need a new code path — we just need to populate the existing one.
- Appending auto-derived pairs into the same `collision_skip_pairs_` member after the user-supplied launch parameter is read keeps the existing INFO log (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp:168-171`) honest — it will report the auto-pairs alongside any user-supplied ones, which is desirable for debugging.
- Order: auto-pair generation must happen **after** `getChain` succeeds (line 112) but can happen anywhere before `buildCollisionObjects` / first goal callback. Cleanest spot is right after the joint-limits parse (after line 142).
- The joint-order check is an isolated 28-line block (lines 375-401). Deletion is local; no other code reads `order_ok` or `order_warn_oss`.

## Decision

**Decision:** modify only `ik_fcl_ompl_planner.cpp`. Single file, two surgical edits:

### Edit 1 — auto-derive adjacent-link skip pairs

After line 142 (end of joint-limits build), before the TRAC-IK construction:

```cpp
// Auto-derive adjacent-link skip pairs from the KDL chain so that
// physically-connected (and therefore geometrically-overlapping) links
// are not flagged as self-collisions. Order in the pair string does not
// matter — isInCollision matches both "a:b" and "b:a".
{
    std::string prev = base_link_;
    for (unsigned int i = 0; i < kdl_chain_right.getNrOfSegments(); ++i) {
        const std::string& cur = kdl_chain_right.getSegment(i).getName();
        collision_skip_pairs_.push_back(prev + ":" + cur);
        prev = cur;
    }
}
```

Effect: with `base_link_ = "torso_link"` and the standard 7-segment right arm chain, 7 entries are appended to `collision_skip_pairs_` (`torso_link:right_shoulder_pitch_link`, ..., `right_wrist_pitch_link:right_wrist_yaw_link`). The existing INFO log block at line 168-171 will report them.

### Edit 2 — delete the joint-order false-positive check

Delete lines 375-401 (the entire `bool order_ok` block including the trailing `RCLCPP_ERROR` and its `if (!order_ok)` wrapper). The directly-following `planning_positions` lookup is by name and survives any joint_state ordering.
