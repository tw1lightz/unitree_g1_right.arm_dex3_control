# Plan 01-07: Fix trajectory-end ramp q=0 jerk

## Why

User-reported regression after Plan 01-06 PASS:
> "空闲态退出没动,让机械臂动起来,结束后它会突然抖动,然后回到原点."

(Idle Ctrl+C: no jerk ✓. Send a goal, let arm move, **at end of trajectory** it jerks then returns to standing.)

This was hidden by Plan 01-04's `gracefulRelease()`: the second jerk that ran after the trajectory-end ramp was much larger (and started from a state where the body had already taken over), so the user perceived a single bad event. After Plan 01-06 deleted `gracefulRelease()`, the smaller pre-existing trajectory-end-ramp jerk became visible on its own.

## Root cause

```@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:235-246
    for (int step = 0; step <= interp_steps; ++step) {
      double t = static_cast<double>(step) / interp_steps;
      double value = (1.0 - t) * 1.0 + t * 0.0; // Linear interpolation
      unitree_hg::msg::LowCmd final_cmd;
      final_cmd.motor_cmd[JointIndex::kNotUsedJoint].q = static_cast<float>(value);
      cmd_pub_->publish(final_cmd);
      rclcpp::sleep_for(sleep_ns);
    }
```

`final_cmd` is default-constructed every iteration → all `motor_cmd[i].q = 0`, `kp = 0`, `kd = 0`. arm_sdk in planner-controlled mode (because for the first frame `kNotUsedJoint.q = 1.0`) appears to apply some internal default kp to the q=0 targets, jerking every joint toward its q=0 reference pose. Then the master switch fades to 0.0 and the body controller takes over. So the user sees: trajectory completes → 1 quick jerk → 1 second of slow recovery to standing.

The trajectory-following path (line 184-200) does it correctly: every `cmd_msg` populates each motor's `q` from `latest_joint_positions_` and sets `kp = 60.0, kd = 1.5`. The fix is to apply the same population pattern to the end-of-trajectory ramp.

## Constraint analysis

- `latest_joint_positions_` is updated by `lowstateCallback` from `/lf/lowstate` whenever a new low-state message arrives. By trajectory end the values reflect the actual physical pose of the arm (with the usual servo-tracking error, which is exactly what the body controller would smoothly absorb).
- `joint_name_to_index` is an external `std::map<std::string, size_t>` from `g1_dex3_joint_defs.hpp`, the same iteration source the trajectory-following loop uses.
- The ramp's frame cadence (50 frames × 20 ms = 1 s) and the `kNotUsedJoint.q` linear fade from 1.0 to 0.0 are unchanged. Only the inside of each frame's `final_cmd` gains the same population block as the trajectory-following frames.
- This is functionally equivalent to publishing a "stand still here, fade authority" command — exactly what the original code was trying (and failing) to do.

## Decision

Single surgical edit in `joint_trajectory_executor.cpp`'s end-of-trajectory ramp (line ~235-246). Each frame:

```cpp
unitree_hg::msg::LowCmd final_cmd;
final_cmd.motor_cmd[JointIndex::kNotUsedJoint].q = static_cast<float>(value);
// Plan 01-07: hold every joint at its current actual position with the
// same kp/kd as the trajectory-following frames, so arm_sdk never sees
// the default kp=0+q=0 fields and never jerks toward the q=0 reference.
for (const auto& pair : joint_name_to_index) {
  size_t idx = pair.second;
  if (latest_joint_positions_.size() > idx) {
    final_cmd.motor_cmd[idx].q = latest_joint_positions_[idx];
  } else {
    final_cmd.motor_cmd[idx].q = 0.0f;
  }
  final_cmd.motor_cmd[idx].dq = 0.f;
  final_cmd.motor_cmd[idx].kp = 60.0f;
  final_cmd.motor_cmd[idx].kd = 1.5f;
  final_cmd.motor_cmd[idx].tau = 0.f;
}
cmd_pub_->publish(final_cmd);
rclcpp::sleep_for(sleep_ns);
```

The added block is byte-for-byte the same population pattern as the trajectory-following path at line 184-192. We do NOT extract a helper because (a) it is two callsites, (b) the line count saving is small, (c) the trajectory-following frames also overwrite `q` with trajectory targets and we would have to thread that distinction through any helper.

## Notes

This bug is in the project since `83b34a5 project_init`. It was masked by the larger jerk in Plan 01-04's `gracefulRelease()` (which Plan 01-06 deleted). It would also have masked Plan 01-04 in reverse: had the original developer noticed the trajectory-end-ramp jerk and decided "we need a graceful release", they might have arrived at exactly the wrong fix shape Plan 01-04 took. Plan 01-07 closes the loop by fixing the original ramp the same way the trajectory-following frames already get it right.
