# Plan 01-10: enable PD control mode by setting motor_cmd[idx].mode = 1

## Why

User report after Plan 01-09:
> "ш┐ШцШпц▓бхПШ."

Plan 01-09 was conceptually correct (drive the arm to standing yourself before the handover). But the user supplied a working reference: `/home/unitree/Desktop/hand_eye_calib/right_arm_mode.py`. That script's `smooth_exit()` does the **same** thing in spirit тАФ fade master while motor_cmd holds the lock pose with PD control тАФ and it works. Comparing the reference's `G1_29_ArmController.__init__` against our `trajectoryCallback` reveals the actual missing field:

```python
# reference: hand_eye_calib/robot_control/robot_arm.py:120
for id in G1_29_JointIndex:
    self.msg.motor_cmd[id].mode = 1
    if id.value in arm_indices:
        ...
        self.msg.motor_cmd[id].kp = self.kp_low  # 80.0
        self.msg.motor_cmd[id].kd = self.kd_low  # 3.0
```

`motor_cmd[id].mode = 1` is set for every joint at controller init, and the same `self.msg` is reused every cycle (so mode=1 sticks). Our `trajectoryCallback` constructs a fresh `unitree_hg::msg::LowCmd` every frame and **never sets the `mode` field on any motor_cmd**. The default value is 0.

In arm_sdk's protocol, `mode` selects the per-motor control mode:
- `mode = 0` тЖТ disabled / idle / fallback (motor does NOT track our q/kp/kd; arm_sdk applies whatever its internal default is, which evidently does cause SOME motion тАФ the arm reaches goal in our trajectory тАФ but does not produce a true stiff hold).
- `mode = 1` тЖТ PD-position control: motor tracks `q` with stiffness `kp` and damping `kd`, plus optional `tau` feed-forward.

**Consequence:** Plans 01-07, 01-08, 01-09 all populated `q + kp + kd` correctly, but with `mode = 0` arm_sdk was treating those values as advisory at best. The end-of-trajectory ramp's "stiff hold" was never actually stiff. When the master switch released, the arm was wherever arm_sdk's mode=0 fallback had drifted to (тЙИ trajectory end-point, because trajectory-following also under-controlled), and the body controller then snapped it to standing on its own servo timeline.

This is the real reason every "make the handover smooth" attempt failed: nothing we did under mode=0 actually committed the arm to a controlled position.

## Constraint analysis

The reference also differs in:
- `kp_low = 80` (we use 60)
- `kd_low = 3.0` (we use 1.5)
- `kp_wrist = 40` (we use 60)
- `tau` carries gravity feed-forward (we use 0)

These are all gain / feed-forward differences, separately diagnosable. The minimal high-signal change is just `mode = 1` because:
1. It is the only difference that toggles a categorical control mode, not a numeric tuning. If `mode = 0` is in fact ignoring our commands, all the populate / interpolate / fade work has been a no-op.
2. If `mode = 1` alone produces smooth return to standing, the kp/kd/tau differences are tuning niceties, not functional requirements.
3. If `mode = 1` is necessary but not sufficient, the residual symptom will tell us which other field (gains vs gravity FF) to adjust next.

We adopt the minimum-variable principle: change exactly one thing, observe.

## Decision

Two single-line insertions in `joint_trajectory_executor.cpp`'s `trajectoryCallback`:

### Edit 1 тАФ trajectory-following loop's per-joint population

In the loop at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:199-210`, add as the first line of the loop body:

```cpp
cmd_msg.motor_cmd[idx].mode = 1;
```

### Edit 2 тАФ end-of-trajectory ramp's per-joint population

In the loop at `@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:269-283`, add as the first line of the loop body:

```cpp
final_cmd.motor_cmd[idx].mode = 1;
```

The destructor's `final_cmd` (one-shot `kNotUsedJoint.q = 0` reaffirmation in shutdown) does NOT need mode=1 because by that point the body controller is already in charge; the LowCmd is just a final master-switch assertion.
