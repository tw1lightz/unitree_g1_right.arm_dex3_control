# Plan 01-09: executor actively interpolates q to standing snapshot

## Why

User reports after Plan 01-08:
> "没有平滑回归,还是瞬间冲击."

(No smooth return; still snap.)

## What Plan 01-08 got wrong

Plan 01-08 was based on the assumption that arm_sdk's `kNotUsedJoint.q` master switch produces a real continuous blend between planner and body authority over the 0.0–1.0 range. With that assumption, fading `kp` while the master switch faded would let the body controller's standing-pose pull take effective influence midway through the ramp.

The user's verification result indicates that assumption is wrong. The most plausible alternative model: **arm_sdk's master switch behaves as a binary threshold** (or close enough that body authority is effectively zero until the switch crosses some boundary near 0.0). Whatever blend happens between planner and body during the fade is not strong enough — or not in the right direction — to pull the arm toward standing while the planner is still asserting kp>0 at the trajectory end-point. So:

- Plan 01-08 has the planner publishing q=current_actual with kp slowly fading from 60 to 0 → arm continues to hold near the trajectory end-point throughout the ramp.
- Body controller's standing pull is suppressed (or simply ignored) until master ≈ 0.0.
- The instant master crosses to 0.0, body has full authority and snaps the arm to standing at its own (fast) servo rate.

## Constraint analysis

The only behavior we genuinely control is what we publish. While the master switch is high (planner has authority), arm_sdk strictly executes our `q` with the `kp/kd` we set. So the robust fix — **independent of any master-switch protocol detail** — is:

> During the ramp, publish a sequence of `q` values that smoothly walks the arm from the trajectory end-point to the standing pose. Keep `kp = 60, kd = 1.5` so the planner stiffly tracks that interpolation. By frame 150, the arm is already at standing pose; whatever the body controller does on takeover is a small holding correction with no visible motion.

This works whether the master switch is binary or blended: the arm physically is at standing by ramp end, so there's nothing for the body controller to snap to.

The only remaining question is **what the standing pose IS**. Three options:

1. **Hard-coded constants.** Not in the codebase; would require us to invent values per joint per robot model.
2. **Launch parameter.** Pushes configuration burden onto the operator.
3. **Snapshot at `trajectoryCallback` entry.** When a trajectory message arrives, the arm is **guaranteed** to be at standing — either because the previous trajectory's end-of-trajectory ramp returned authority and the body controller settled it back to standing, or because this is the first trajectory after launch and the body controller has been holding standing since boot. The current `latest_joint_positions_` (lowstate-driven) at that exact moment is, by definition, the standing pose. **Free, accurate, model-agnostic.**

We pick (3).

A subtle invariant: between trajectoryCallback entry and the start of the end-of-trajectory ramp, the arm has moved through every waypoint. We must capture the standing snapshot at entry, BEFORE the trajectory-following loop scrambles `latest_joint_positions_` along the way.

## Decision

Three surgical edits to `trajectoryCallback` in `joint_trajectory_executor.cpp`:

### Edit 1 — snapshot standing pose at callback entry

Right after the early-return guards and side-detection (just after the `hand_cmd_pub` selection), capture:

```cpp
// Plan 01-09: snapshot standing pose at callback entry. Arm is guaranteed
// to be at standing here (either previous trajectory's ramp settled it
// back, or robot just booted with body controller holding standing). This
// snapshot is the target the end-of-trajectory ramp will drive toward.
std::vector<float> standing_pose;
if (!latest_joint_positions_.empty()) {
  standing_pose = latest_joint_positions_;
}
```

### Edit 2 — snapshot trajectory end-point just before the ramp

Right before `const double interp_duration`:

```cpp
// Plan 01-09: snapshot the trajectory end-point as the ramp's starting
// pose. We interpolate from this fixed start (not the live, drifting
// latest_joint_positions_) toward the standing snapshot.
std::vector<float> ramp_start_positions;
if (!latest_joint_positions_.empty()) {
  ramp_start_positions = latest_joint_positions_;
}
```

### Edit 3 — replace ramp body to interpolate q + revert Plan 01-08's kp/kd fade

In the ramp's inner population loop, replace the current body with:

```cpp
final_cmd.motor_cmd[idx].q =
  (ramp_start_positions.size() > idx && standing_pose.size() > idx)
    ? static_cast<float>((1.0 - t) * ramp_start_positions[idx] + t * standing_pose[idx])
    : ((latest_joint_positions_.size() > idx)
        ? latest_joint_positions_[idx]
        : 0.0f);
final_cmd.motor_cmd[idx].dq = 0.f;
final_cmd.motor_cmd[idx].kp = 60.0f;  // Plan 01-09: revert Plan 01-08 fade; stiff servoing tracks the interpolated path
final_cmd.motor_cmd[idx].kd = 1.5f;   // Plan 01-09: revert Plan 01-08 fade
final_cmd.motor_cmd[idx].tau = 0.f;
```

Frame 0 (`t = 0`) → `q = ramp_start_positions[idx]` = trajectory end-point. Identical to Plan 01-07's frame-0 q (matches arm's actual position) — no jerk. Frame 150 (`t = 1`) → `q = standing_pose[idx]`. arm is at standing pose with kp=60 stiff hold. Master switch reaches 0.0 at exactly this moment; body takes over an arm that's already at standing.

## Notes — risk model for the snapshot approach

The standing snapshot relies on `trajectoryCallback`'s precondition: arm is at standing pose at callback entry. Two ways this could be violated:

1. **Two trajectories arrive back-to-back without enough delay for the previous ramp to complete and the body to settle.** In that case, `standing_pose` would capture some intermediate pose, and the next trajectory's ramp would interpolate to that wrong target. Currently the `JointTrajectory` topic is single-publisher (planner) and trajectories naturally serialize — should not happen in practice. If it does, executor would visibly converge to a slightly-off "standing" but next trajectory would re-snapshot a real standing pose, self-correcting after one cycle.
2. **`latest_joint_positions_` is empty at callback entry.** Only possible if no `/lf/lowstate` message has arrived yet — extremely brief window after node start. The fallback path uses `q = 0.0f` for that joint — not jerk-causing because frame 0 still has `q = ramp_start_positions[idx]` (also potentially empty → also 0.0f), so frame 0 publishes (kp=60, q=0). This is the only case where the original Plan 01-07-era q=0 behavior could manifest, and only on a malformed first message. Acceptable risk.

Both risks are documented in the SUMMARY's `key-decisions` for future reference.
