## Task 3: Velocity-based time parameterization in trajectory construction

<read_first>
- src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp (lines 633-644 for existing trajectory construction loop, line 232 for velocity_limits_)
</read_first>

<action>
Replace the fixed-time trajectory construction loop (lines 637-644) with velocity-based timing:

1. Keep `path.interpolate()` and `const auto& states = path.getStates()` unchanged.
2. Build trajectory points with per-segment dt:
   - For each consecutive pair `states[idx-1]` → `states[idx]`, compute `dt = max_over_joints(|Δq_i| / (velocity_limits_[planning_joints[i]] * velocity_scale_))`, clamped to at least `min_time_step_`. For joints not in `velocity_limits_`, use fallback velocity 10.0 rad/s.
   - First point (idx=0): use `min_time_step_` as dt from implicit start.
   - Accumulate `cumulative_time += dt` for each point's `time_from_start`.
3. Log total trajectory duration: `RCLCPP_INFO(... "Trajectory: %zu waypoints, %.3f seconds total duration", ...)`.

Key: `planning_joints` (local variable in goalPoseCallback) maps index i to joint name for velocity_limits_ lookup.
</action>

<acceptance_criteria>
- Each consecutive waypoint pair has dt computed from max joint displacement / (velocity limit × velocity_scale)
- dt is at least min_time_step for every segment
- time_from_start is cumulative (monotonically increasing)
- Trajectory total duration is logged
- Joints without velocity limits use fallback 10.0 rad/s
- No reference to time_step_ anywhere in the loop
</acceptance_criteria>

## Task 4: Pre-publish trajectory validation with auto-fix

<read_first>
- src/unitree_g1_dex3_stack-main/src/ik_fcl_ompl_planner.cpp (lines 646-656 for current publish block, line 232 for joint_limits_ and velocity_limits_)
</read_first>

<action>
Insert a validation block between trajectory construction and `traj_pub_->publish(traj_msg)`:

1. **Position validation**: For each point, for each joint, check position is within `joint_limits_[planning_joints[i]]`. If out of range:
   - Auto-fix: clamp to [lower, upper]
   - Log: `RCLCPP_WARN(... "Auto-fix: clamped joint %s position %.4f to [%.4f, %.4f]", ...)`
   - Set `position_fixed = true`

2. **Velocity validation**: For consecutive points, compute implied velocity `|Δq| / dt`. If exceeds `velocity_limits_[joint] * velocity_scale_`:
   - Auto-fix: recompute dt for that segment as `max_over_joints(|Δq_i| / (vel_limit_i * velocity_scale_))`, clamped to min_time_step_. Recompute cumulative time_from_start for all subsequent points.
   - Log: `RCLCPP_WARN(... "Auto-fix: stretched dt at waypoint %zu for velocity limit", ...)`
   - Set `velocity_fixed = true`

3. **Re-validate** after fixes. If still failing, `RCLCPP_ERROR(... "Trajectory validation failed after auto-fix, rejecting")` and `return` (do not publish).

4. If validation passes (with or without fixes), log: `RCLCPP_INFO(... "Trajectory validation passed%s", fixed ? " (with auto-fix)" : "")`.

5. Then proceed to existing `traj_pub_->publish(traj_msg)`.
</action>

<acceptance_criteria>
- Position violations are auto-fixed by clamping, logged with RCLCPP_WARN
- Velocity violations are auto-fixed by stretching dt, logged with RCLCPP_WARN
- Failed validation after auto-fix produces RCLCPP_ERROR and trajectory is NOT published
- Passed validation produces RCLCPP_INFO mentioning "(with auto-fix)" if fixes were applied
- The existing traj_pub_->publish and "Plan published" log remain in place after validation
</acceptance_criteria>
