# Plan 01-04: Graceful arm release on executor shutdown

## Why

User flagged that during live verification, if the executor terminal is killed mid-trajectory (or even between trajectories), the current destructor publishes a single `q = 0.0` LowCmd which **instantly** removes planner control. Combined with the moderate joint stiffness (`kp = 60.0`, `kd = 1.5`), this creates a perceivable mechanical jerk and is unsafe.

Acceptable behaviour: when the executor process exits, the arm should **smoothly** transfer control back to the robot body so the body controller can naturally return to the standing pose.

## Background (from `joint_trajectory_executor.cpp`)

The unitree `arm_sdk` LowCmd protocol uses `motor_cmd[JointIndex::kNotUsedJoint].q` as a **control-authority signal**:
- `1.0` → planner has full authority (executor follows trajectory).
- `0.0` → release; robot's body controller resumes ownership and returns the arm to the standing/default pose.

The existing `trajectoryCallback` already does a 1.0 → 0.0 ramp (50 steps × 20 ms = 1 s) at the **end of every trajectory** (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:210-221`). Mid-trajectory exit is the missing case.

The destructor (`@/home/unitree/Desktop/unitree_dex3/src/unitree_g1_dex3_stack-main/src/joint_trajectory_executor.cpp:103-110`) publishes a single instantaneous `q = 0.0`:

```cpp
~JointTrajectoryExecutor() override {
  RCLCPP_INFO(this->get_logger(), "Shutting down Joint Trajectory Executor Node");
  unitree_hg::msg::LowCmd final_cmd;
  final_cmd.motor_cmd[JointIndex::kNotUsedJoint].q = 0.0f;
  cmd_pub_->publish(final_cmd);
}
```

## Constraint analysis

The destructor cannot reliably publish during a graceful ramp because by the time the destructor runs, `rclcpp::shutdown()` has already invalidated the context (`rclcpp`'s built-in SIGINT handler fires on Ctrl+C, calls shutdown, then `spin()` returns). After shutdown, `cmd_pub_->publish(...)` will throw.

The clean ROS 2 fix is to **disable rclcpp's default signal handler**, install our own, and run a manual spin loop. When the flag is set, we exit the loop cleanly **before** rclcpp shutdown is called, then perform the 3-second ramp while the publisher is still valid, and finally invoke `rclcpp::shutdown()`.

## Decision

**Decision:** modify only `joint_trajectory_executor.cpp`.

- Disable rclcpp's built-in SIGINT/SIGTERM handlers via `rclcpp::uninstall_signal_handlers()` immediately after `rclcpp::init()` (Foxy API; the `SignalHandlerOptions` `init()` overload only exists from Galactic onward, verified at `/opt/ros/foxy/include/rclcpp/utilities.hpp`).
- Install a process-level `std::atomic<bool> g_shutdown_requested` SIGINT/SIGTERM handler that just sets the flag (signal-safe).
- Replace `rclcpp::spin(node)` in `main()` with a `SingleThreadedExecutor` loop that polls the flag and calls `spin_some(50ms)` until the flag fires.
- Add a public `JointTrajectoryExecutor::gracefulRelease()` method: 150 steps × 20 ms = 3 s linear interpolation of `LowCmd.motor_cmd[kNotUsedJoint].q` from 1.0 down to 0.0, with `rclcpp::ok()` short-circuit at each step.
- After the spin loop exits, call `node->gracefulRelease()` then `rclcpp::shutdown()`.
- **Keep** the existing destructor's single-`q=0.0` publish as a last-resort fallback (for cases where the process is forcibly killed before reaching `gracefulRelease`).

User-selected duration: **3 seconds (150 steps × 20 ms)**.

## Tasks

### Task 1 — Implement graceful release path (`joint_trajectory_executor.cpp`)

- Add includes: `<csignal>`, `<atomic>`.
- Add file-scope `std::atomic<bool> g_shutdown_requested{false}` and a static `signal_handler` function.
- Add public `void gracefulRelease()` method on `JointTrajectoryExecutor`: 3-second linear ramp 1.0 → 0.0 of `kNotUsedJoint.q`, with `rclcpp::ok()` short-circuit and a leading INFO log.
- Modify `main()`:
  - `rclcpp::init(argc, argv, rclcpp::InitOptions(), rclcpp::SignalHandlerOptions::None);`
  - `std::signal(SIGINT, signal_handler); std::signal(SIGTERM, signal_handler);`
  - Replace `rclcpp::spin(node)` with a `SingleThreadedExecutor` polling loop checking `g_shutdown_requested && rclcpp::ok()`.
  - Call `node->gracefulRelease()` after the loop returns, before `rclcpp::shutdown()`.
- Leave the destructor's `q = 0.0` publish in place (last-resort safety net for forced kills).

- The destructor still has its `q = 0.0` publish as fallback (do not regress).
- `RCLCPP_INFO` for "Graceful release" is logged once per shutdown.
